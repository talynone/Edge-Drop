## Memory Optimization Plan for Edge-Drop

**Goal:** Significantly reduce RAM usage with **zero** visual or functional changes. All optimizations preserve existing behavior exactly; they only cut wasted memory/CPU.

---

### Phase 1 — Process elimination (biggest wins: ~80–120 MB)

**1.1 Remove the dead "detector" BrowserWindow entirely** (`electron/main/window.ts`)
- Delete `detectorWindow`, `createDetectorWindow()`, `getDetectorBounds()`, and all references in `repositionWindow()`, `setHeartbeatPaused()`, and the heartbeat interval.
- Why it's safe (per the window's own comments + code review): opening is driven by `startCursorPoll()` which reads `screen.getCursorScreenPoint()` directly; the detector is click-through, has no external consumers, and is self-described as a "no-op drag-event absorber." Drag-in already works via the cursor poll → edge dwell → `setInteractive(true)` → drop lands on the main window.
- Removes one entire Chromium renderer process (~80–120 MB) plus its preload, GPU context, and compositor surface.

**1.2 Strip 3 unused dependencies from `package.json`**
- Remove `@radix-ui/react-scroll-area`, `@radix-ui/react-separator`, `svg-path-bbox` — grep confirms **zero imports** anywhere in `src/` or `electron/`. They're externalized into the packaged app as dead weight. Keep `@resvg/resvg-js` (used by drag icons).

---

### Phase 2 — Kill the always-on hot loops (idle CPU + memory churn)

**2.1 Adaptive cursor poll** (`electron/main/window.ts` `startCursorPoll`)
- Replace the fixed `setInterval(…, 16)` (60 Hz forever) with a self-rescheduling `setTimeout` whose delay adapts to state:
  - **Idle** (cursor far from edge AND panel closed): ~250 ms (4 Hz) — enough to catch an approach.
  - **Near edge** (cursor within ~450 px of the stuck edge, OR panel open): ~16 ms (60 Hz) — unchanged responsiveness for opening/closing.
- Transition is automatic: near-edge detection already exists in the poll body (`nearEdge` variable); reuse it to pick the next delay.
- The only consumer (`useEdgeHover.ts`) already tolerates variable cadence (it's event-driven with timers), so no renderer change needed.

**2.2 Adaptive always-on-top heartbeat** (`electron/main/window.ts`)
- Stretch the heartbeat from a fixed 500 ms to ~3000 ms when idle. Fullscreen-app suppression is still handled (the heartbeat re-asserts on focus, on `ready-to-show`, on drag-end via `setHeartbeatPaused(false)`, and on `setVisible(true)`), so the panel still snaps back when exiting fullscreen — just the *redundant* every-half-second reassertion is slowed.
- Keep `setHeartbeatPaused` semantics intact (drag-out still pauses/resumes correctly, just without the detector window branch).

---

### Phase 3 — Stop the memory leaks & growth

**3.1 Remove the runaway `drag_debug.txt` logging** (`electron/main/drag.ts`)
- Delete `logDrag()` and all ~17 call sites. The file is already **440 KB and growing**; each call does synchronous `appendFileSync` on the drag hot path (disk I/O + string alloc per drag). Replace `console.log('[DragDebug] …')` usage with nothing in release.
- Delete the existing `drag_debug.txt` and `clipboard_debug.log` from the repo root, and add them to `.gitignore` (already covered by `*.log`; add `drag_debug.txt` explicitly).

**3.2 Bound and invalidate the main-process caches** (`electron/store/ItemStore.ts`, `electron/main/drag.ts`)
- `dataUrlCache`, `fileEntryCache`, `fileDataUrlCache`: lower flush thresholds and, critically, **invalidate entries when their owning item is deleted** (`delete`, `clearUnpinned`, `pruneExpired`, `split`, `removeSubitem`) so base64 image strings don't outlive their items. Currently a deleted image's data URL can linger until 200+ *other* items accumulate.
- `iconCache` in `drag.ts`: already capped at 64 — leave as is.

**3.3 Silence release-build logging** (`electron/main/*.ts`, `electron/main/window.ts`)
- Wrap the ~32 `console.log`/`console.error` calls in main behind `if (APP_CONFIG.is.dev)` (config already exposes `is.dev`), and gate the renderer `console-message` forwarder in `window.ts` the same way. Stops per-IPC string formatting + stdout writes in the packaged app. Dev behavior unchanged.

---

### Phase 4 — Reduce committed GPU/renderer memory

**4.1 Lazy-load onboarding videos** (`src/Onboarding.tsx`)
- Change the 6 `<video src="placeholder_*.mp4">` to load on demand: only set the `src` for the **currently visible** slide (lazy mount the `<video>` element per active slide, not all at once). Keeps first-run onboarding visually identical, but Chromium no longer preloads/indexes all 37 MB of videos into the renderer origin cache at once.
- Pure renderer change; no main-process impact; videos stay in `public/`.

**4.2 `content-visibility: auto` on offscreen clipboard items** (`src/styles/item.css`)
- Add `content-visibility: auto; contain-intrinsic-size: 0 88px;` to `.item`. Chromium skips rendering/holding textures for list items scrolled out of view, shrinking GPU memory for long histories. Visually identical (the panel already virtualizes via scroll). Reverts cleanly if any jank appears.

---

### Phase 5 — Build hygiene (smaller asar, fewer files tracked)

**5.1 Delete 7 stray `electron.vite.config.*.mjs` temp files** in the repo root (left by crashed dev runs) and add the pattern to `.gitignore`.

---

### What I will NOT change (to honor "no functional/visual change")
- The panel's look, animations, blur, flares, clip-path transitions.
- Clipboard capture, paste, drag-out, merge/split, pinning, incognito logic.
- Settings, tray menu, multi-monitor, stick left/right behavior.
- The onboarding window's existence (only its video loading becomes lazy).

### Verification plan after implementation
1. `npm run typecheck` passes.
2. `npm run build` succeeds; confirm `out/main/index.js` no longer references `createDetectorWindow`.
3. Manual smoke test in `npm run dev`: hover-open from edge, copy text/image/files, drag-out to Explorer, pin/delete, open settings, toggle incognito, trigger onboarding — all behave exactly as before.
4. Observe RAM in Task Manager: expect the main process tree to drop by roughly one renderer process (~80–120 MB) plus lower steady-state idle CPU.
